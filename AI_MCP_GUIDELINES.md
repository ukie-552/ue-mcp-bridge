# AI使用MCP规则指南

本文档为AI助手使用Unreal Engine MCP工具时的重要规则和注意事项。请务必仔细阅读并遵守这些规则，以确保引擎稳定性和项目安全。

---

## 一、操作频率与批量限制

### 1.1 单次操作数量限制

**危险操作**：一次性执行过多MCP操作可能导致引擎崩溃或无响应。

| 操作类型 | 建议单次上限 | 风险等级 |
|---------|------------|---------|
| 创建Actor | ≤ 50个 | 中 |
| 创建蓝图节点 | ≤ 100个 | 中 |
| 创建材质节点 | ≤ 80个 | 中 |
| 创建UMG控件 | ≤ 30个 | 高 |
| 删除资源 | ≤ 20个 | 高 |
| 修改属性 | ≤ 200个 | 低 |
| Sequencer关键帧 | ≤ 500个 | 中 |

### 1.2 操作间隔建议

```
推荐操作间隔：
- 低风险操作：无强制间隔
- 中风险操作：每10次操作后等待100-200ms
- 高风险操作：每5次操作后等待300-500ms
- 批量操作：建议分批执行，每批之间等待500ms以上
```

### 1.3 禁止的批量操作组合

以下操作组合在同一时间段内执行风险极高：

- 同时创建大量Actor + 同时修改其属性
- 批量删除资源 + 批量创建同名资源
- 大量蓝图节点操作 + 编译蓝图
- 多个Niagara系统同时创建和修改

## 二、UMG控件命名规则

### 2.1 重名控件警告

**严重警告**：在同一个Widget Blueprint中创建重名控件会导致内存泄露！

```
错误示例：
- 在同一Widget中创建两个名为 "Button_Start" 的按钮
- 重命名控件时使用已存在的名称

正确做法：
- 每个控件使用唯一名称
- 命名遵循 "类型_功能" 格式，如：Btn_Start, Txt_Title, Img_Background
- 重命名前先检查目标名称是否已存在
```

### 2.2 推荐的UMG命名规范

| 控件类型 | 前缀 | 示例 |
|---------|-----|------|
| Button | Btn_ | Btn_Play, Btn_Settings |
| TextBlock | Txt_ | Txt_Title, Txt_Score |
| Image | Img_ | Img_Background, Img_Icon |
| CanvasPanel | Cvs_ | Cvs_Root, Cvs_Overlay |
| VerticalBox | Vbx_ | Vbx_MenuList |
| HorizontalBox | Hbx_ | Hbx_ButtonRow |
| GridPanel | Grd_ | Grd_Inventory |
| ScrollBox | Scr_ | Scr_Content |
| Border | Bdr_ | Bdr_Container |
| ProgressBar | Prb_ | Prb_Health |
| Slider | Sld_ | Sld_Volume |
| CheckBox | Chk_ | Chk_Fullscreen |
| EditableText | Edt_ | Edt_PlayerName |
| ComboBox | Cbx_ | Cbx_Quality |

### 2.3 布局坐标系统规则（重要）

**核心原则**：子控件的Position是相对于其直接父控件的局部坐标系，而非屏幕坐标或根控件坐标。

#### 2.3.1 坐标系统理解

```
Widget层级结构示例：
RootCanvas (0, 0) ────────────────────────
    │
    ├── Panel_Menu (100, 50) ─────────────
    │       │
    │       ├── Btn_Option (20, 10) ──────  ← 实际屏幕位置计算：
    │       │                                  RootCanvas(0,0) 
    │       └── Txt_Title (200, 30)           + Panel_Menu(100,50)
    │                                          + Btn_Option(20,10)
    └── Panel_HUD (0, 0)                      = 屏幕位置(120, 60)
            │
            └── Txt_Score (50, 20)
```

#### 2.3.2 常见陷阱

**问题**：直接获取子控件的坐标可能得到 (0, 0) 或非预期值

```
错误理解：
❌ 认为子控件的Position是屏幕绝对坐标
❌ 在根父级坐标系中直接设置子控件位置
❌ 忽略中间父控件的坐标偏移

正确理解：
✓ 子控件的Position始终相对于其直接父控件
✓ 移动控件时需要在正确的父级坐标系内操作
✓ 复杂层级结构需要逐级计算实际位置
```

#### 2.3.3 位置设置最佳实践

| 场景 | 正确做法 |
|------|---------|
| 移动子控件 | 在其直接父控件的坐标系内设置Position |
| 精确定位 | 先了解完整的控件层级结构 |
| 批量移动 | 考虑移动父控件而非逐个移动子控件 |
| 跨层级定位 | 使用Anchors和Alignment辅助定位 |

#### 2.3.4 锚点(Anchors)的影响

锚点设置会显著影响Position的计算方式：

```
锚点位置与Position的关系：

1. 锚点在左上角 (0, 0)：
   - Position从父控件左上角计算
   - 最直观，推荐初学者使用

2. 锚点在中心 (0.5, 0.5)：
   - Position从父控件中心计算
   - 适合居中显示的元素

3. 锚点拉伸模式：
   - Position表示偏移量
   - 控件大小由锚点范围决定

建议：设置位置前先确认锚点配置
```

#### 2.3.5 布局操作检查清单

在修改控件位置/布局前，AI应当：

1. **确认控件层级**：使用 `get_widget_controls` 获取完整层级结构
2. **识别直接父控件**：确定Position的参考坐标系
3. **检查锚点设置**：理解当前锚点对Position的影响
4. **计算正确坐标**：在正确的父级坐标系内设置位置
5. **验证结果**：操作后确认控件出现在预期位置

### 2.4 UMG操作安全检查清单

在执行UMG相关操作前，AI应当：

1. 检查目标Widget Blueprint是否存在
2. 获取现有控件列表，确认无重名
3. 验证父控件是否存在（如果指定了parent_control）
4. 确认控件类型是否有效
5. 操作完成后验证结果

## 三、蓝图操作注意事项

### 3.1 蓝图编译时机

- 创建或修改多个节点后，不应立即强制编译
- 建议完成一系列相关操作后再编译
- 编译失败时，应先检查节点连接是否完整

### 3.2 节点连接规则

- 连接前验证源引脚和目标引脚类型兼容
- 避免创建循环依赖的连接
- 执行引脚(Exec pins)只能连接到执行引脚
- 数据引脚类型必须匹配或可转换

### 3.3 变量操作安全

- 删除变量前检查是否被引用
- 重命名变量会自动更新引用，但仍需验证
- 创建变量时避免使用保留名称（如 "self", "world", "owner"）

---

## 四、材质系统操作规则

### 4.1 材质节点限制

- 单个材质图表节点数建议不超过150个
- 复杂材质应使用Material Function模块化
- 避免过深的节点连接链（建议深度≤10层）

### 4.2 材质实例创建

- 创建Material Instance前确保父材质存在
- 参数名称必须与父材质定义一致
- 批量创建材质实例时注意内存占用

---

## 五、Sequencer操作规则

### 5.1 关键帧批量操作

- 单次添加关键帧不超过500个
- 多轨道操作时按轨道分批执行
- 长序列建议分段处理

### 5.2 轨道管理

- 添加轨道前确认Actor已添加到序列
- 删除轨道时注意相关关键帧的清理
- 避免在播放状态下修改序列

---

## 六、Niagara系统操作规则

### 6.1 发射器和模块创建

- 单个Niagara System发射器数量建议≤10个
- 模块添加遵循正确的执行顺序
- 参数设置需符合模块预期类型

### 6.2 性能考量

- 复杂粒子系统会显著影响性能
- 测试时建议降低粒子数量
- 避免同时预览多个复杂Niagara系统

---

## 七、资源管理规则

### 7.1 资源路径规范

```
有效路径格式：/Game/Folder/SubFolder/AssetName
示例：/Game/Characters/Hero/BP_HeroCharacter

禁止使用：
- 相对路径
- 不存在的路径
- 特殊字符（空格、中文等）
```

### 7.2 资源删除安全

- 删除前检查资源是否被引用
- 使用GetAssetReferences工具验证引用关系
- 删除操作不可逆，需谨慎执行

### 7.3 资源重命名和移动

- 重命名/移动会自动更新引用
- 操作后验证引用是否正确更新
- 避免与现有资源重名

---

## 八、场景操作规则

### 8.1 Actor操作

- 大量Actor操作时建议分批执行
- SpawnActor后等待引擎稳定再进行属性修改
- 删除Actor前确认无重要引用

### 8.2 Level操作

- 创建新Level后需要正确设置World Settings
- LoadLevel操作会卸载当前Level
- 保存Level前确保无未完成的操作

---

## 九、错误处理与恢复

### 9.1 常见错误码

| 错误类型 | 含义 | 建议处理 |
|---------|------|---------|
| ASSET_NOT_FOUND | 资源不存在 | 检查路径是否正确 |
| INVALID_PARAM | 参数无效 | 验证参数类型和值 |
| OPERATION_FAILED | 操作失败 | 检查引擎状态，重试 |
| TIMEOUT | 操作超时 | 减少批量大小，重试 |
| CRASH_RISK | 崩溃风险 | 立即停止，等待引擎恢复 |

### 9.2 异常恢复流程

1. 操作失败时，等待至少1秒
2. 检查引擎响应状态
3. 如引擎无响应，建议用户重启编辑器
4. 恢复后从失败点继续，但减少批量大小

## 十一、紧急情况处理

### 11.1 引擎无响应

如果发现引擎长时间无响应：

1. 立即停止所有MCP操作
2. 等待30秒观察是否恢复
3. 如未恢复，建议用户通过任务管理器结束UE5进程
4. 重启后检查项目完整性

### 11.2 内存占用过高

如果内存占用异常：

1. 停止创建新资源
2. 建议用户执行垃圾回收（gc命令）
3. 考虑分多次会话完成操作
4. 建议用户保存并重启编辑器
